# PitchForge — Claude Code Context

## What This Project Is
PitchForge is a LangGraph-based AI pipeline that takes a job posting (from Upwork, Freelancer, etc.) and produces a personalized, high-quality proposal. It uses RAG to pull relevant experience from a local profile, scores job fit, generates a draft, critiques and refines it in a loop, then outputs a final approved proposal.

---

## Stack
- Python 3.11
- LangGraph — orchestration
- Gemini 2.5 Flash — LLM for all nodes
- Google gemini-embedding-2-preview — embeddings for RAG
- ChromaDB (local, persistent) — vector store for profile
- LangChain Google GenAI — LLM + embeddings wrapper
- uv — dependency management

## State Schema (state.py)
```python
class PitchforgeState(TypedDict):
    job_posting: str           # Raw job input
    job_analysis: dict         # Structured job breakdown (Node 1 output)
    profile_matches: list      # RAG chunks from ChromaDB (Node 2 output)
    client_info: Optional[str] # Client research (not yet built)
    proposal_draft: str        # Current draft (Node 4 output)
    critic_feedback: str       # Critic's feedback (Node 5 output)
    iteration_count: int       # Loop counter — max 3
    quality_score: int         # Critic's score 0-100
    fit_score: int             # Job fit score (Node 3 output)
    suggested_price: str       # Pricing recommendation (Node 3 output)
    matched_skills: list       # Skills from job that match profile (Node 3 output)
    missing_skills: list       # Skills from job missing from profile (Node 3 output)
    clarifying_questions: list # Questions for client
    final_proposal: str        # Approved final output
    should_apply: bool         # Human decision after seeing fit score
    human_approved: bool       # Human checkpoint flag
    human_feedback: str        # Feedback text from human on rejection (Node 6 output)
    is_human_revision: bool    # True after first human rejection — caps critic loop to 1 pass
```

## Graph Flow (Full)
```
analyzer → retriever → scorer → fit_checkpoint
                                      ↓
                             [interrupt: apply? y/n]
                              /               \
                            "n"               "y"
                             ↓                 ↓
                            END            generator
                                               ↓
                                            critic
                                               ↓
                              is_human_revision OR score>=70 OR iter>=3?
                               /                                \
                             YES                                NO
                               ↓                                ↓
                        human_checkpoint                    generator
                               ↓                           (critic loop)
                  [interrupt: "y" approve or type feedback]
                   /                         \
                 "y"                    feedback text
                   ↓                         ↓
                  END             human_feedback saved
                              is_human_revision=True
                                             ↓
                                         generator
                                             ↓
                                          critic
                                             ↓
                              is_human_revision=True → human_checkpoint
                                        ← loop repeats ←
```

---

## Critic-Generator Loop
- Max 3 iterations before escalating to human checkpoint
- On auto-revision passes, generator reads `critic_feedback` and fixes specific issues
- `is_human_revision=True` permanently short-circuits the critic loop — after any human rejection, every subsequent critic pass routes directly to human_checkpoint (no more looping)

---

## Conditional Edge Routing

```python
# After fit_checkpoint (Node 3.5)
should_apply=False → END
should_apply=True  → generator

# After critic (Node 5)
is_human_revision OR quality_score>=70 OR iteration_count>=3 → human_checkpoint
otherwise → generator

# After human_checkpoint (Node 6)
human_approved=True  → END
human_approved=False → generator  (feedback stored in human_feedback)
```

---

## Human Checkpoint Behavior (Node 6)
Single `interrupt()` — user either types `y` to approve, or types feedback text to reject and request changes.

- `"y"` → `human_approved=True` → END
- any other text → `human_approved=False`, `human_feedback=<text>`, `is_human_revision=True` → generator

Generator prioritizes `human_feedback` over `critic_feedback` when building revision context.

## Backend API (runner.py)

| Function | What it does |
|----------|-------------|
| `run_analysis(job_input)` | Starts new thread, runs graph to fit_checkpoint interrupt, returns fit report |
| `run_generation(thread_id, should_apply)` | Resumes fit_checkpoint with y/n, runs to human_checkpoint, returns proposal draft |
| `run_revise(thread_id, feedback)` | Resumes human_checkpoint with feedback text, runs one revision pass, returns updated draft |
| `run_finalize(thread_id)` | Resumes human_checkpoint with "y", runs to END, returns final proposal |

## Python Packaging (Monorepo)
`backend/` imports `pitchforge` as a package. This works via:
- `pitchforge/__init__.py` — makes it a proper package
- `pitchforge/pyproject.toml` — setuptools with `where = [".."]` so editable install adds `PitchForge/` to sys.path
- `backend/pyproject.toml` — `[tool.uv.sources]` with `pitchforge = { path = "../pitchforge", editable = true }`
- All internal imports use `from pitchforge.X import` (not bare `from X import`)

## Rules for This Project
1. Build and test one node at a time before moving to the next
2. Never modify state.py schema without checking all existing nodes still work
3. All nodes return only the keys they write — never return full state
4. Keep node files single-responsibility — one node per file
5. graph.py owns all LangGraph wiring — nodes, edges, checkpointer
6. main.py is only the entry point — collects input, runs graph, handles interrupts
7. runner.py owns all LangGraph invocations for the backend — no graph calls outside it
