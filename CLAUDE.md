# PitchForge — Claude Code Context

## What This Project Is
PitchForge is a LangGraph-based AI pipeline that takes a job posting (from Upwork, Freelancer, etc.) and produces a personalized, high-quality proposal. It uses RAG to pull relevant experience from a local profile, scores job fit, generates a draft, critiques and refines it in a loop, then outputs a final approved proposal.

---

## Stack
- Python 3.11
- LangGraph — orchestration
- Gemini 2.5 Flash — LLM for all nodes
- Google gemini-embedding-2-preview — embeddings for RAG
- ChromaDB (local, persistent) — vector store for profile
- LangChain Google GenAI — LLM + embeddings wrapper
- uv — dependency management

## State Schema (state.py)
```python
class PitchforgeState(TypedDict):
    job_posting: str           # Raw job input
    job_analysis: dict         # Structured job breakdown (Node 1 output)
    profile_matches: list      # RAG chunks from ChromaDB (Node 2 output)
    client_info: Optional[str] # Client research (Node 4 — not built yet)
    proposal_draft: str        # Current draft (Node 4 output)
    critic_feedback: str       # Critic's feedback (Node 5 output)
    iteration_count: int       # Loop counter — max 3
    quality_score: int         # Critic's score 0-100
    fit_score: int             # Job fit score (Node 3 output)
    suggested_price: str       # Pricing recommendation (Node 3 output)
    clarifying_questions: list # Questions for client
    final_proposal: str        # Approved final output
    should_apply: bool         # Human decision after seeing fit score
    human_approved: bool       # Human checkpoint flag
```

## The Critic-Generator Loop (Core Logic)
```
Node 4 (Generator)
        ↓
Node 5 (Critic)
        ↓
   quality_score >= 70?
   /              \
 YES               NO + iteration_count < 3
   ↓                        ↓
Node 6               Back to Node 4
(Human checkpoint)   (with critic_feedback injected)
```
- Max 3 iterations — prevents infinite loop
- On revision passes, generator reads critic_feedback and fixes specific issues
- This loop is the reason LangGraph is used — clean conditional edges

---

## Conditional Edge Logic
```python
# After Node 3 — human reviews fit score
fit_score < 40               → show report, auto-cancel → END
fit_score >= 40              → show report, prompt [y/N]
  human says N               → END
  human says Y (should_apply)→ Node 4 (generate proposal)

# After Node 5 — is the draft good enough?
quality_score >= 70 OR iteration_count >= 3 → Node 6 (human checkpoint)
quality_score < 70 AND iteration_count < 3  → Node 4 (revise)
```

## Rules for This Project
1. Build and test one node at a time before moving to the next
2. Never modify state.py schema without checking all existing nodes still work
3. All nodes return only the keys they write — never return full state
4. Keep node files single-responsibility — one node per file
5. graph.py owns all LangGraph wiring — nodes, edges, checkpointer
6. main.py is only the entry point — collects input, runs graph, handles interrupts

## How to Run
```bash
# One-time setup (from pitchforge/)
cd pitchforge
uv run setup_rag.py

# Run the LangGraph pipeline directly
uv run main.py

# Run FastAPI backend (from backend/, once built)
cd backend
uvicorn app.main:app --reload
```
